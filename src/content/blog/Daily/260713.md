# 람다 함수 (Lambda Function)

이름 없는 짧은 함수를 즉석에서 만드는 `lambda`를 초보 수준부터 중수 수준까지 다룬다. 기본 문법과 `def`와의 차이부터 시작해서, 실무에서 자주 쓰는 패턴(정렬 키, `map`/`filter`, 클로저)과 흔히 걸려 넘어지는 함정까지 정리한다.

## 흐름

1. **람다 기본 문법** — `lambda`가 무엇이고 `def`와 어떻게 다른지 확인한다.
2. **정렬·필터링에서 활용** — `sorted()`, `map()`, `filter()`와 함께 쓰는 법을 익힌다.
3. **클로저와 함정 이해** — 람다가 바깥 변수를 어떻게 참조하는지, 반복문 안에서 왜 자주 실수하는지 확인한다.
4. **언제 쓰지 말아야 하는지 판단** — 가독성을 해치는 경우를 구분한다.

## 1. 람다 기본 문법 — 이름 없는 함수 만들기

### 왜 배우는가
파이썬에서 함수를 만들 때 보통 `def`를 쓰지만, "딱 한 번, 아주 짧게" 쓰고 버릴 함수를 매번 이름 붙여 정의하는 건 번거롭다. `lambda`는 이런 상황을 위해 만든 "한 줄짜리 이름 없는 함수" 문법이다. 초보자에게는 "함수도 값처럼 다룰 수 있다"는 감각을 처음 잡아주는 문법이고, 중수에게는 `sorted`, `map` 같은 함수에 짧은 로직을 즉석에서 끼워 넣는 실무 도구다.

### 상세 설명
- 문법은 `lambda 매개변수들: 표현식` 형태다.
- `lambda` 키워드 뒤에 매개변수를 쓰고(여러 개면 쉼표로 구분), 콜론(`:`) 뒤에 반환할 표현식을 하나만 적는다.
- **문법적으로 표현식 하나로 제한된다.** `if`, `for`, 여러 줄의 로직, `print()` 같은 문장(statement)은 람다 안에 넣을 수 없다.
- 별도로 `return`을 쓰지 않아도 콜론 뒤의 표현식 결과가 자동으로 반환된다.
- 람다도 함수 객체이므로, 변수에 저장하거나 다른 함수의 인자로 바로 넘길 수 있다.

```python
add = lambda a, b: a + b
add(3, 5)
```

**코드 읽는 순서**
1. `lambda a, b: a + b`가 실행되면, `a`와 `b`를 받아서 `a + b`를 반환하는 함수 객체 하나가 만들어진다.
2. 이 함수 객체는 `add`라는 이름에 대입된다.
3. `add(3, 5)`를 호출하면 `a=3`, `b=5`로 `3 + 5`가 계산되어 `8`이 반환된다.

### `def`로 쓴 것과 비교

같은 동작을 `def`로 쓰면 이렇게 된다.

```python
def add(a, b):
    return a + b
```

두 코드는 완전히 같은 함수를 만든다. 차이는 문법의 길이와 이름 유무뿐이다. `lambda`는 `def`의 편의 문법일 뿐, 새로운 기능이 아니다.

## 2. `def`와 `lambda`, 언제 무엇을 쓰나

### 왜 배우는가
람다에 익숙해지면 뭐든 람다로 줄여 쓰고 싶은 유혹이 생긴다. 하지만 람다는 "아주 짧고, 한 번 쓰고 버리는" 로직에 어울리는 도구이지, `def`를 완전히 대체하는 도구가 아니다. 이 구분을 못 하면 오히려 코드가 더 읽기 어려워진다.

### 상세 설명
| 상황 | 추천 |
|---|---|
| 로직이 표현식 하나로 끝난다(계산, 비교, 간단한 변환) | `lambda` |
| 함수에 이름을 붙여서 여러 곳에서 재사용하고 싶다 | `def` |
| 조건문, 반복문, 여러 줄의 처리가 필요하다 | `def` |
| 함수를 다른 함수의 인자 자리에 "즉석에서" 넘기고 싶다 | `lambda` |
| 독스트링으로 함수 설명을 남기고 싶다 | `def` (람다는 독스트링을 붙이기 애매하다) |

### 예제 스니펫

```python
numbers = [3, 1, 4, 1, 5, 9, 2, 6]

# 짧은 로직 하나를 그 자리에서만 쓰는 경우 -- lambda가 자연스럽다
squaredOnce = list(map(lambda n: n ** 2, numbers))

# 로직이 복잡해지거나 재사용이 필요한 경우 -- def가 낫다
def classifyNumber(n):
    if n % 2 == 0:
        return 'even'
    elif n > 5:
        return 'odd-large'
    else:
        return 'odd-small'

classified = [classifyNumber(n) for n in numbers]
```

`classifyNumber`처럼 조건이 여러 개면 람다로는 아예 표현할 수 없다(표현식 하나로 제한되기 때문이다). 이런 경우는 처음부터 `def`를 쓰는 것이 맞다.

## 3. `sorted()`의 `key`로 쓰기 — 가장 흔한 실무 활용

### 왜 배우는가
리스트를 정렬할 때 "어떤 기준으로 정렬할지"를 매번 별도 함수로 정의하는 건 번거롭다. `sorted()`와 `list.sort()`는 `key` 매개변수에 함수를 하나 받는데, 이 자리가 람다가 가장 자주 쓰이는 곳이다.

### 상세 설명
- `sorted(리스트, key=함수)`를 쓰면, 리스트의 각 항목을 그 함수에 넣은 결과를 기준으로 정렬한다.
- 원래 값이 아니라 "함수를 통과시킨 결과"로 비교하기 때문에, 튜플의 특정 위치나 딕셔너리의 특정 키를 기준으로 정렬할 때 특히 유용하다.
- `reverse=True`를 함께 쓰면 내림차순으로 정렬한다.

```python
employees = [
    {'name': '민호', 'age': 34},
    {'name': '유나', 'age': 27},
    {'name': '도윤', 'age': 41},
]

sortedByAge = sorted(employees, key=lambda person: person['age'])
sortedByNameLength = sorted(employees, key=lambda person: len(person['name']), reverse=True)
```

**코드 읽는 순서**
1. `sorted(employees, key=lambda person: person['age'])`는 `employees`의 각 딕셔너리에서 `'age'` 값만 뽑아서 그 값을 기준으로 비교한다.
2. 결과적으로 `sortedByAge`는 나이가 어린 순서(`27, 34, 41`)로 정렬된 딕셔너리 리스트가 된다.
3. `sortedByNameLength`는 이름 길이가 긴 순서로 정렬되며, `reverse=True`이므로 내림차순이다.

## 4. `map()`, `filter()`와 함께 쓰기

### 왜 배우는가
"리스트의 모든 항목을 똑같이 변환하고 싶다"거나 "조건에 맞는 항목만 골라내고 싶다"는 요구는 매우 흔하다. `map()`과 `filter()`는 이런 작업을 반복문 없이 표현하게 해주는 내장 함수이고, 그 안에 넣는 짧은 변환·조건 로직 자리가 람다의 대표적인 쓰임새다.

### 상세 설명
- `map(함수, 이터러블)`은 이터러블의 각 항목에 함수를 적용한 결과를 순서대로 돌려주는 이터러블을 만든다.
- `filter(함수, 이터러블)`은 함수의 결과가 참(`True`)인 항목만 남긴 이터러블을 만든다.
- 두 함수 모두 결과 자체는 이터러블(map 객체, filter 객체)이라서, 눈으로 확인하려면 `list()`로 감싸야 한다.
- 실무에서는 같은 일을 리스트 컴프리헨션으로도 표현할 수 있는데, 파이썬 커뮤니티에서는 간단한 경우 리스트 컴프리헨션이 더 읽기 쉽다는 의견이 많다. 다만 `map`/`filter` + `lambda` 조합도 여전히 널리 쓰인다.

```python
prices = [12000, 8500, 25000, 4300, 17800]

# map: 모든 가격에 10% 할인을 적용한다
discounted = list(map(lambda p: round(p * 0.9), prices))

# filter: 만원 이상인 가격만 남긴다
expensiveOnly = list(filter(lambda p: p >= 10000, prices))

# 같은 일을 리스트 컴프리헨션으로 표현하면
discountedAlt = [round(p * 0.9) for p in prices]
expensiveOnlyAlt = [p for p in prices if p >= 10000]
```

**코드 읽는 순서**
1. `map(lambda p: round(p * 0.9), prices)`는 `prices`의 각 값을 순서대로 람다에 넣어서 `p * 0.9`를 반올림한 값을 하나씩 만들어낸다.
2. `list(...)`로 감싸면 이 결과들이 실제 리스트로 만들어진다.
3. `filter(lambda p: p >= 10000, prices)`는 `p >= 10000`이 참인 값만 통과시킨다. `list(...)`로 감싸면 `[12000, 25000, 17800]`이 나온다.

## 5. `functools.reduce()`와 함께 쓰기

### 왜 배우는가
리스트 전체를 하나의 값으로 누적해서 합치는 작업(총합, 최댓값, 문자열 이어붙이기 등)이 필요할 때가 있다. `reduce()`는 이런 "누적" 로직을 표현하는 함수이고, 누적 방식을 람다로 짧게 정의하는 경우가 많다.

### 상세 설명
- `reduce()`는 표준 라이브러리 `functools` 모듈에 있어서 `from functools import reduce`로 가져와야 한다.
- `reduce(함수, 이터러블)`은 이터러블의 앞 두 항목에 함수를 적용하고, 그 결과와 다음 항목에 다시 함수를 적용하는 식으로 계속 누적해서 최종적으로 값 하나를 만든다.
- 단순 합계나 최댓값 정도는 `sum()`, `max()`가 더 간단하다. `reduce()`는 "곱셈 누적"처럼 내장 함수로 바로 되지 않는 누적 로직에 주로 쓴다.

```python
from functools import reduce

quantities = [2, 3, 4, 5]
totalProduct = reduce(lambda acc, n: acc * n, quantities)
```

**코드 읽는 순서**
1. 처음에 `acc=2`, `n=3`으로 시작해서 `2 * 3 = 6`을 계산한다.
2. 그 결과 `6`이 새 `acc`가 되고, 다음 항목 `n=4`와 곱해져서 `24`가 된다.
3. `24`가 다시 `acc`가 되고, 마지막 항목 `n=5`와 곱해져서 `120`이 된다.
4. 더 이상 항목이 없으므로 `120`이 `totalProduct`에 최종 대입된다.

## 6. 조건부 표현식과 결합하기

### 왜 배우는가
람다는 표현식 하나만 담을 수 있어서 `if` 문을 그대로 못 쓴다. 하지만 파이썬의 "조건부 표현식"(삼항 표현식)은 그 자체가 하나의 표현식이라서 람다 안에 넣을 수 있다. 이걸 알면 람다로 표현할 수 있는 범위가 꽤 넓어진다.

### 상세 설명
- 조건부 표현식의 문법은 `참일때값 if 조건 else 거짓일때값`이다.
- 이 전체가 표현식 하나이므로, 콜론 뒤에 그대로 넣을 수 있다.
- 조건이 여러 단계로 겹치면(중첩 삼항 표현식) 가독성이 급격히 떨어지므로, 이 시점에서는 `def`로 옮기는 것이 낫다.

```python
classify = lambda n: 'even' if n % 2 == 0 else 'odd'
classify(7)   # 'odd'
classify(10)  # 'even'
```

## 7. 람다와 클로저 — 바깥 변수를 기억하는 함수 만들기

### 왜 배우는가
람다(그리고 일반 함수)는 자신이 정의된 스코프의 변수를 계속 참조할 수 있다. 이 성질을 이용하면 "특정 값을 미리 고정해둔 함수"를 함수로 찍어낼 수 있다. 이런 함수를 클로저(closure)라고 부르며, 설정값을 미리 넣어둔 함수를 여러 개 만들어야 할 때 유용하다.

### 상세 설명
- 함수 안에서 `lambda`를 만들면, 그 람다는 바깥 함수의 지역 변수를 계속 기억한다.
- 바깥 함수가 이미 실행을 끝내고 사라진 뒤에도, 람다는 그 변수 값을 계속 사용할 수 있다.

```python
def makeTaxCalculator(taxRate):
    return lambda price: round(price * (1 + taxRate))

vatCalculator = makeTaxCalculator(0.1)
localTaxCalculator = makeTaxCalculator(0.05)

vatCalculator(10000)     # 11000
localTaxCalculator(10000)  # 10500
```

**코드 읽는 순서**
1. `makeTaxCalculator(0.1)`을 호출하면, `taxRate=0.1`을 기억하는 람다 함수 하나가 만들어져서 `vatCalculator`에 저장된다.
2. `makeTaxCalculator(0.05)`를 별도로 호출하면, 이번에는 `taxRate=0.05`를 기억하는 또 다른 람다가 만들어져서 `localTaxCalculator`에 저장된다.
3. 두 함수는 서로 다른 `taxRate` 값을 각자 독립적으로 기억하고 있으므로, 같은 `10000`을 넣어도 결과가 다르다.

## 8. 흔한 함정 — 반복문 안에서 만든 람다는 "나중에" 변수를 읽는다

### 왜 배우는가
람다를 반복문 안에서 여러 개 만들다 보면, 모든 람다가 똑같은 값을 참조하는 당황스러운 상황을 자주 겪는다. 이 문제의 원인을 정확히 알아야 실무에서 이런 버그를 피할 수 있다.

### 상세 설명
- 람다(그리고 클로저 전반)는 변수의 "그 순간 값"을 저장하는 것이 아니라, 변수 자체를 계속 참조한다. 그래서 람다가 실제로 호출되는 시점에 그 변수가 가진 최신 값을 읽는다.
- 반복문에서 매번 새 람다를 만들어도, 반복문이 다 돌고 난 뒤에는 그 변수가 마지막 값 하나만 가지고 있으므로, 모든 람다가 똑같이 그 마지막 값을 참조하게 된다.
- 해결 방법은 매개변수의 기본값으로 그 시점의 값을 "고정"해서 넘기는 것이다. 기본값은 함수(람다 포함)가 정의되는 시점에 한 번만 계산되기 때문에, 이 방법으로 그 순간의 값을 확실히 붙잡아둘 수 있다.

```python
# 문제가 있는 코드
multipliers = []
for i in range(1, 4):
    multipliers.append(lambda x: x * i)

[m(10) for m in multipliers]
# [30, 30, 30] -- 셋 다 마지막 i값인 3을 참조해서 예상과 다르게 나온다
```

```python
# 고친 코드: 기본값으로 그 순간의 i를 고정한다
fixedMultipliers = []
for i in range(1, 4):
    fixedMultipliers.append(lambda x, i=i: x * i)

[m(10) for m in fixedMultipliers]
# [10, 20, 30] -- 각 람다가 만들어질 때의 i값을 각자 따로 기억한다
```

**코드 읽는 순서**
1. 첫 번째 코드에서는 `lambda x: x * i`가 반복문을 세 번 돌면서 세 번 만들어지지만, 셋 다 똑같이 바깥의 `i`라는 이름을 참조할 뿐, 각자 다른 값을 저장해두지 않는다.
2. 반복문이 끝나면 `i`는 마지막 값인 `3`이 되고, 세 람다 모두 호출되는 시점에 이 `3`을 읽어와서 결과가 전부 `30`이 된다.
3. 고친 코드에서는 `lambda x, i=i: x * i`처럼 `i=i`라는 기본값을 추가했다. 기본값은 람다가 만들어지는 바로 그 순간의 `i` 값으로 고정되므로, 반복문의 각 회차마다 서로 다른 값이 각 람다에 개별적으로 저장된다.

## 9. 람다를 쓰지 않는 게 나은 경우

### 왜 배우는가
람다에 익숙해질수록 "짧게 쓸 수 있는데 왜 def를 쓰지"라는 생각이 들 수 있다. 하지만 짧다고 항상 좋은 코드는 아니다. 언제 람다를 피해야 하는지 아는 것도 실력이다.

### 상세 설명
- **이름 있는 함수를 굳이 람다로 만들어서 변수에 대입하는 것은 권장하지 않는다.** 파이썬 스타일 가이드(PEP 8)에서도 이런 경우 `def`를 쓰라고 안내한다. `def`로 만들면 함수 이름 자체가 에러 메시지나 디버깅 정보에 그대로 남지만, 람다를 변수에 대입하면 이런 정보가 `<lambda>`로만 표시되어 디버깅이 불편해진다.

```python
# 권장하지 않는 방식
square = lambda x: x ** 2

# 권장하는 방식
def square(x):
    return x ** 2
```

- 조건이 여러 겹으로 중첩되거나, 여러 줄의 로직이 필요하면 람다로 억지로 구겨 넣지 말고 `def`로 옮긴다. 람다 안의 로직이 한눈에 안 들어오기 시작하면 이미 늦은 신호다.
- 같은 람다를 여러 곳에서 반복해서 쓰고 있다면, 그 자체가 "이름을 붙여서 재사용해야 한다"는 신호다.

## 전체 핵심 요약

| 상황 | 문법 | 한 줄 요약 |
|---|---|---|
| 짧은 표현식 하나로 끝나는 함수 | `lambda 매개변수: 표현식` | 이름 없이 즉석에서 만드는 함수다 |
| 정렬 기준을 지정할 때 | `sorted(리스트, key=lambda ...)` | 비교할 값을 함수로 뽑아낸다 |
| 모든 항목을 변환할 때 | `map(lambda ..., 이터러블)` | 각 항목에 함수를 적용한다 |
| 조건에 맞는 항목만 남길 때 | `filter(lambda ..., 이터러블)` | 함수 결과가 참인 항목만 남긴다 |
| 값을 하나로 누적할 때 | `reduce(lambda ..., 이터러블)` | 앞 결과와 다음 항목을 계속 합친다 |
| 람다 안에서 분기가 필요할 때 | `A if 조건 else B` | 조건부 표현식으로 표현식 하나 안에 분기를 담는다 |
| 값을 미리 고정한 함수가 필요할 때 | 함수 안에서 `lambda` 반환 | 클로저로 바깥 변수를 기억하게 만든다 |
| 반복문 안에서 람다를 여러 개 만들 때 | `lambda x, i=i: ...` | 기본값으로 그 순간의 값을 고정해야 한다 |
| 람다에 이름을 붙여서 재사용하고 싶을 때 | `def 함수이름(...):` | 이럴 땐 애초에 `def`를 쓰는 것이 맞다 |
